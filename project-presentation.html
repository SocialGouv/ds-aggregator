Solution actuelle

Repos

Les repos ds-* ont pour ambition d’être des briques réutilisables pour n’importe quel administration utilisatrice de DS

  • ds-collector : instance docker/Node dédiée à la synchro des données DS pour une procédure donnée. expose une API pour en extraire des données (stats + validity) (composant non couplé à wif)
  • ds-dashboard : dashboard statique qui expose des stats (composant non couplé à wif)

Spécifique WIF

  • wif-bo-public : instance docker/Node qui expose les stats et un endpoint pour le validity check. (données sous forme de fichiers statiques via un volume docker)
  • wif-extractor : scripts spécfiques WiF pour extraire les données stats + validity dans un volume docker

Problèmes à résoudre

Le ds-collector actuel ne sait récupérer qu’une procédure DS. Or dans notre cas, on gère plusieurs démarches (+30, 2 par UD), ce qui génère autant d’instances et fichiers de données à aggréger pour produire les stats. Il serait donc plus judicieux de centraliser ces données pour un token DS donné. c.a.d. que le collecteur pourrait récupérer toutes les démarches accessibles à un token.

De plus, l’extracteur produit actuellement des fichiers statiques JSON (régulièrement via un CRON) qui sont partagés avec l’API publique via un volume docker, ce qui est problématique pour notre future infra. Deux BDD, une privée (synchros) et une publique (exports stats+validity) seraient plus appropriées.

La centralisation des données permettra de constuire plus facilement des UI/dashboards pour les agents instructeurs et les administrations.

Le plus gros du travail est donc de refactorer le code de ds-collector (ou de partir from scratch) pour gérer un backend de stockage plus scalable et de modifier les exports de données en fonction.

A prendre en compte

Il faut pouvoir isoler les données privées et sensibles (issues des synchros DS) des données que l’on expose à l’API publique (acces aux stats + validity)

Les données qu’on récupère de l’API DS sont schema-less : c’est un gros JSON variable selon les procédures, et dans lequel on doit pouvoir naviguer pour extraire certaines stats.

Les volumes de données ne sont pas énormes (5000 actuellement) et on aura potentiellement une obligation de destruction de données au bout d’un certain temps donc pas de souci de scalabilité de données.

On ne l’a pas exploité pour l’instant car on utilise des tâches CRON pour récupérer les dossiers, mais DS propose un webhook déclenché à chaque mise à jour d’un dossier

Parallèlement Christophe Dumont (qu’on a pas encore rencontré :)) fait un refresh du site frontend de présentation qui redirige vers les procédures DS. Lorsqu’une nouvelle démarche est créée, l’intrapreneur (Daniel Balmy) la rajoute sur le site (ca se fera surement dans un fichier de config json). En général je récupère ces infos à la main et configure les ds-collector et dashboard pour ces nouvelles démarches. On pourra voir si on peut automatiser une liaison entre les deux :) cross-project pipelines FTW !

Recos

ds-collector

Créer un service capable de :

  • synchro N démarches DS à partir d’un token et d’une liste de procédures (sauf erreur l’API DS ne permet pas de discover les démaraches dispos…)
  • stocker les démarches dans une base dédiée
  • exposer une API pour les stats et données

wif-extractor

Créer les scripts spécifiques WIF pour extraire les stats et données utiles au validity check et les stocker dans une BDD dédiée plutot qu’un volume docker.

Exemple de données de stats extraite pour une démarche donnée : https://wif-dashboard.num.social.gouv.fr/stats/16075.json
